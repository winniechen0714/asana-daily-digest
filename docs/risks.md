## 2026-03-03

### RISK-001 — 修改任務數量上限（已知，暫不影響）

**發現時機：** 執行效能分析時

**描述：**
`search_modified_tasks` 使用 Asana Search API，`limit` 設定為 100（每頁上限）。
`asana_get` 已實作分頁（pagination），當結果超過 100 筆時會自動補抓，但每頁多一輪 API 呼叫，stories 查詢也會等比例增加。

**影響範圍：**
- 若某日修改任務數超過 100 筆 → 執行時間線性拉長
- 每多 100 筆任務 ≈ 多 100 次 stories API 呼叫 ≈ 多約 20s 執行時間
- 目前購案商機專案的日常修改量約 100 筆，尚在第一頁範圍內

**現況：**
觀察中，不需立即處理。若未來日常修改量持續超過 100 筆，考慮以下優化方向：
1. 在 stories 篩選前加入任務層級的預篩條件，減少無效 API 呼叫
2. 改用 Asana Audit Log API（需 Business/Enterprise 方案）直接查詢事件，跳過逐任務掃描的設計

**相關程式碼：** [digest.py:63-72](../digest.py)

### RISK-002 — Asana Search API 排序不保證涵蓋全部近期活動（已知，低風險）

**發現時機：** 資料驗證過程

**描述：**
Asana Search API 依 `modified_at` 排序時，回傳順序由 API 決定，不保證「最近修改」的任務一定排在前 100 筆。若某個高頻修改任務佔去名額，部分低頻但有效變動的任務可能在分頁較後面出現。

**影響範圍：**
- 在 `asana_get` 有正確實作分頁的前提下，所有任務最終都會被處理，不會遺漏
- 風險僅存在於 `asana_get` 分頁邏輯失效時（如 API 回傳 `next_page` 格式異動）

**現況：**
已有分頁保護，低風險。定期人工抽查 Slack 摘要與 Asana 實際活動是否吻合即可。

**相關程式碼：** [digest.py:29-47](../digest.py)

### RISK-003 — GitHub Actions cron 延遲（已知，時間範圍已修正）

**發現時機：** 排程設定確認；2026-03-04 首次實際觀測到嚴重延遲

**描述：**
GitHub Actions 的 `schedule` 觸發在 UTC 00:00 整點為全平台最高峰，**實際延遲可達 2 小時以上**（非原先預估的 ±15 分鐘）。
2026-03-04 實際觀測：設定 UTC 00:00 觸發，實際執行時間為 UTC 02:24（台灣時間 10:24），延遲約 **144 分鐘**。

**影響範圍（修正前）：**
- 傳送時間延後至 10:24，而非預期的 08:00
- 時間範圍因採用 rolling window（執行當下 -24h），導致掃描區間跟著偏移至 10:23 前日 ~ 10:23 當日，並非預期的 08:00~08:00

**處置（2026-03-04 已修正）：**
1. 時間範圍計算改為固定錨點：無論腳本幾點執行，`since` / `until` 均固定為台灣時間「昨日 06:00」至「今日 06:00」。
2. cron 從 `0 0 * * *`（UTC 00:00）改為 `0 22 * * *`（UTC 22:00 = 台灣 06:00），避開 UTC 00:00 整點高峰，實測延遲通常降至 1–5 分鐘以內。
若未來仍需「確保 10 分鐘內送達」的 SLA，可改用外部 cron 服務（如 cron-job.org）呼叫 GitHub API 的 `workflow_dispatch` 端點觸發。

**相關程式碼：** [digest.py:270-281](../digest.py)

### RISK-004 — Slack Webhook 無回傳確認機制（已知，低風險）

**發現時機：** 程式碼審查

**描述：**
目前 `send_to_slack` 僅確認 HTTP 200 + `"ok"` 回應，無法得知訊息是否實際顯示在頻道中（如 Webhook 被靜音、頻道已封存等情境）。

**影響範圍：**
- 腳本執行成功但 Slack 頻道看不到訊息，不會自動告警

**現況：**
低風險。若需更強的可靠性，可改用 Slack Bot Token + `chat.postMessage` API，該 API 回傳完整的訊息物件與錯誤碼。目前不必要。

**相關程式碼：** [digest.py:238-246](../digest.py)
